Q1 · CSKG

The five main Common-Sense Knowledge Graphs — what each one is and how it’s built. Common Sense KB deck · slides 10–104 السؤال الأول: عدّد شبكات المعرفة العامة الخمس واشرح كل وحدة وكيف بُنيت.

unmarked

PromptList the 5 main Common-Sense Knowledge Graphs (CSKGs) from the lecture and explain in one short paragraph what each is and how it is built. Also know the 4 dimensions used to classify them.

عدّد الشبكات الخمس الرئيسية للمعرفة العامة (Common-Sense KGs)، اشرح بإيجاز ما هي كل واحدة وكيف بُنيت، وكن جاهزاً للمقارنة بينها وفق الأبعاد الأربعة في المحاضرة.

— try to list the five yourself first / حاول قبل الإجابة.

First — what does “common sense” even mean?

Common sense is sound practical judgement about everyday matters — the basic background knowledge that nearly all humans share but is hard to put in words. The classic motivating task in the deck is the “dog  |  frisbee  |  catch  |  throw” example: given four common concepts, a system should be able to combine them into a meaningful sentence — and that requires knowing things like dogs play with frisbees, throw is the opposite of catch, etc.

المعرفة العامة (Common Sense): هي المعرفة البدهية المشتركة بين البشر عن الحياة اليومية. مثلاً: الكلب يلعب بالفريسبي، الرمي عكس الإمساك، إلخ. الإنسان يستخدم هذه المعرفة بشكل طبيعي، لكن الحاسوب يحتاج إلى تمثيلها بشكل صريح في شبكة معرفية ليفهم اللغة والسياق.
  • المثال الافتتاحي بالمحاضرة: تعطي النموذج المفاهيم dog | frisbee | catch | throw فيلزمه ربطها في جملة منطقية.
  • هذا غير ممكن دون شبكة معرفية تخبره أن الكلب يستطيع اللعب، وأن الفريسبي قرص طائر، إلخ.

And one extra cross-cutting distinction: top-down (experts encode the rules, e.g. Cyc) vs bottom-up (the system learns from text, e.g. NELL).

The five main CSKGs

1

Cyc / OpenCyc

Lenat 1984 · OpenCyc 4.0 (2012)

A huge, hand-curated commonsense KB built by domain experts — a top-down approach. Symbolic logical assertions, highly structured ontology. Brittle and expensive: classic “expert-system era”.

قاعدة معرفية ضخمة بُنيت يدوياً من قبل خبراء (طريقة top-down). تمثيل رمزي منطقي. مشكلتها: هشّة، مكلفة، ولا تتوسع بسهولة.
top-down · expert-built · logical.
2

ConceptNet

Liu & Singh 2004 · Speer 5.5 (2017)

A multilingual semantic network of everyday concepts (dog — CapableOf — bark). Built by crowdsourcing (Open Mind Common Sense, games like Verbosity) plus Wiktionary, DBpedia, WordNet, OpenCyc. Available as Linked Open Data (JSON-LD API).

شبكة دلالية متعددة اللغات للمفاهيم اليومية، بُنيت عبر الحشد العام (crowdsourcing) — مساهمات الناس وألعاب مثل Verbosity. متاحة كبيانات مفتوحة مرتبطة (LOD).
crowdsourced multilingual graph of concepts.
3

ATOMIC

Sap et al. 2019

An inferential CSKG of if-then knowledge about everyday events: causes and effects across 9 inference dimensions (e.g. PersonX wants…, feels…, as a result PersonY…). Nodes are natural-language phrases, not symbolic concepts. ~300k event nodes, ~880k if-Event-then-* triples.

شبكة معرفة استنتاجية عن الأحداث الاجتماعية والأسباب والنتائج (إن-حدث-إذاً). العقد عبارة عن جمل بلغة طبيعية، وتغطي ٩ أبعاد استدلالية (كنوايا الشخص، ومشاعره، وما يحدث للآخرين بنتيجة فعله).
if-then events, social/causal, natural language.
4

COMET

Bosselut et al. 2019

Not a static graph — a neural Transformer fine-tuned on ATOMIC/ConceptNet that generates new commonsense triples on demand. It solves the fact that symbolic CSKGs are always incomplete: input plain language, get back contextualized knowledge.

ليست شبكة جامدة بل نموذج عصبي (Transformer) دُرّب على ATOMIC و ConceptNet ليولّد ثلاثيات معرفة جديدة عند الطلب. يحل مشكلة أن الشبكات الرمزية ناقصة دائماً.
learned generative CSKG — knowledge on demand.
5

NELL

Carlson 2010 · Mitchell 2015

The Never-Ending Language Learner has been reading the Web continuously since 2010, extracting facts and refining its own ontology. A bottom-up approach. ~100M candidate beliefs, ~3M high-confidence facts, ~3k predicates.

نظام يقرأ الإنترنت منذ ٢٠١٠ ولا يتوقف، يستخرج الحقائق ويحسّن منهجه ذاتياً. منهج bottom-up (من البيانات إلى المعرفة). ١٠٠ مليون اعتقاد، ٣ مليون حقيقة موثوقة، ٣٠٠٠ علاقة.
self-learning, web-scale, never stops.

Three more you should be able to name (bonus)

  • WebChild — automatically extracted fine-grained relations (hasTaste, hasShape, fasterThan, smallerThan); ~18M assertions, ~2M concepts.
  • Wikidata / Wikidata-CS — the commonsense slice of Wikidata, distilled using 3 principles: P1 concepts not entities (“houses have rooms”, not “Versailles has 700 rooms”), P2 common concepts (high corpus frequency), P3 general-domain relations. Result: ~71k nodes / ~102k edges (0.01% of full Wikidata).
  • Visual Genome — scene-graph CSKG from 108,077 images: ~3.8M objects, ~26 attributes/image, ~21 relationships/image, all canonicalized to WordNet senses.
الإضافات الثلاث:
  • WebChild: مستخرَجة آلياً، تركّز على علاقات دقيقة (الطعم، الشكل، الأسرع من…).
  • Wikidata-CS: استخلاص الجانب «البديهي» من ويكي داتا وفق ثلاثة مبادئ: مفاهيم لا كيانات، مفاهيم شائعة، علاقات عامة.
  • Visual Genome: شبكة معرفة بصرية مبنية من ١٠٨ ألف صورة موصوفة بمشاهد كاملة.

One-liners to memorise before the exam

  • Cyc → expert, top-down, logical. خبراء، رمزي.
  • ConceptNet → crowdsourced concept graph. حشد، متعدد اللغات.
  • ATOMIC → if-then social events in natural language. أحداث + استنتاج.
  • COMET → neural generator of CSKG knowledge. شبكة عصبية تولّد المعرفة.
  • NELL → never-ending web extractor. متعلم لا يتوقف.
Q2 · Semantic Web

Merging two graphs about the same book — step by step. Semantic Web deck · slides 17–34 · the canonical exam exercise السؤال الثاني: ادمج شبكتين عن نفس الكتاب (مكتبة إنكليزية A ومكتبة فرنسية F) وأظهر استعلاماً يصبح ممكناً بعد الدمج فقط.

unmarked

PromptYou have two simple bookstore datasets about the same book — English (A) and French (F). Show the full pipeline: from raw tables → RDF graphs → merged graph → adding ontology glue → asking a query that only works after the merge.

لديك جدولان عن نفس الكتاب: مكتبة إنكليزية (A) ومكتبة فرنسية (F). اشرح كل خطوات الدمج: من الجداول الخام إلى رسم بياني RDF لكل مجموعة، ثم الدمج، ثم إضافة معرفة إضافية (glue)، ثم استعلام يعمل بعد الدمج فقط.

— this is the long one; take your time / تأخذ وقتها.

Why this matters

The whole point of the Semantic Web is that two organizations can publish data independently, and a third party can combine them to answer questions neither could answer alone — provided they agree on URIs (universal names) and a tiny bit of shared vocabulary.

الفكرة المركزية: الويب الدلالي يسمح لمنظمتين أن تنشرا بياناتهما بشكل مستقل، ثم يقوم طرف ثالث بـدمج الرسمين البيانيين والإجابة عن أسئلة لم يكن أي منهما قادراً على الإجابة عنها بمفرده. الشرط الوحيد:
  • أن يستخدما عناوين URI عالمية (مثل ISBN) ليتم التعرّف على نفس المورد في الشبكتين.
  • أن نضيف قليلاً من «الصمغ» (glue ontology) عند الدمج لربط المصطلحات المتشابهة.

كل خلية من الجدول صارت إما عقدة (URI أو حرفية literal) أو سهماً معنوناً. النقطة الأساسية: العقدة المرسومة بـالذهبي المنقّط في الرسم الفرنسي هي نفس العقدة الموجودة في الرسم الإنكليزي، لأنها تحمل نفس الـ ISBN. لاحظ أن العلاقات اسمها f:auteur هنا و a:author هناك — هذا فرق سنحلّه لاحقاً.

Merge: the two graphs touch on the shared URI — automatically. الدمج: الرسمان يلتقيان عند الـ URI المشترك تلقائياً.

You don’t need to write any merge code. Because URIs are global identifiers on the Web, an RDF store sees <isbn/000651409X> in both graphs and treats it as one and the same node. The arcs from both sides simply attach to it.

اضغط الأزرار أعلى الرسم لترى كيف ينمو الرسم. لاحظ بنية العقد العمياء (blank nodes) كما في المحاضرة:
  • الناشر _:p عقدة عمياء لها خاصيتان: a:p_name → "Harper Collins" و a:city → "London".
  • المؤلف/auteur _:auth عقدة عمياء واحدة تستقبل كلا السهمين a:author و f:auteur، ولها a:name و f:nom إلى نفس النص "Ghosh, Amitav"، و a:homepage إلى صفحته.
  • المترجم _:tr عقدة عمياء مستقلة لها f:nom → "Besse, Christianne".
ثم:
  • (١) الدمج الأساسي — الرسمان يلتقيان عند الـ ISBN المشترك.
  • (٢) إضافة «الصمغ»: المؤلف والمترجم كلاهما من النوع foaf:Person (r:type).
  • (٣) ربط مع DBpedia: العقدة dbpedia:Amitav_Ghosh ترتبط بالاسم وبالصفحة وبـ foaf:Person.

A query that only works after the merge. استعلام مستحيل قبل الدمج، ممكن بعده.

The user of dataset F asks: « donnes-moi la page d’accueil de l’auteur de l’original » — “give me the home-page of the author of the original of the French book.”

F alone doesn’t know the home-page (it’s in A). A alone doesn’t know which French book points to it. Only the merged graph + the glue triple a:author owl:equivalentProperty f:auteur can answer it.

PREFIX a: <http://example.org/a/>
PREFIX f: <http://example.org/f/>
SELECT ?homepage
WHERE {
  ?frenchBook  f:original   ?origBook .   # the French book → English original
  ?origBook    a:author     ?author .     # the English book → its author
  ?author      a:homepage   ?homepage .   # the author → her/his homepage
}

Returns http://www.amitavghosh.com.

المثلثات الثلاث في الاستعلام:
  1. اذهب من الكتاب الفرنسي إلى الكتاب الأصلي عبر f:original.
  2. من الكتاب الأصلي إلى مؤلفه عبر a:author.
  3. من المؤلف إلى صفحته الشخصية عبر a:homepage.
هذه السلسلة تعبر بين الرسمين، ولم تكن ممكنة قبل الدمج.

Why we still needed the glue, and how to extend further with DBpedia. لماذا نحتاج للصمغ، وكيف نوسّع باستخدام DBpedia.

A bare merge doesn’t know that a:author and f:auteur mean the same thing — they’re just labels chosen by different communities. The glue ontology is three small triples:

a:author         owl:equivalentProperty  f:auteur .
"Ghosh, Amitav"  rdf:type                foaf:Person .
<isbn/000651409X> a:author                ?p .   # so ?p is also f:auteur

Once the author node is typed foaf:Person, you can link it to DBpedia via owl:sameAs / w:reference — and now your dataset reaches all of Ghosh’s other books, his birthplace Kolkata, geo-coordinates, etc.

الخلاصة الكبرى: الويب الدلالي = (URIs مشتركة) + (قليل من الـ glue ontology) + (إمكانية الربط مع DBpedia) ⇒ تكامل البيانات على مستوى الويب. هذا تماماً ما يفعله مستخدمو الإنترنت يومياً، لكن بأسلوب يصلح للآلة.

The minimal cheat-sheet for the exam answer

  1. Map each table to a labelled directed graph (RDF triples).
  2. Use a global identifier (ISBN URI) as the book’s name.
  3. Merge auto-happens on shared URIs.
  4. Add a tiny glue ontology (equivalentProperty, rdf:type).
  5. Now write SPARQL that crosses both graphs.
  6. Extend to DBpedia via owl:sameAs for free knowledge.
Q3 · RDF

RDF/XML ↔ graph — how to convert each into the other. RDF deck · slides 1–86 · the bread-and-butter exercise السؤال الثالث: التحويل بين رسم RDF و RDF/XML — في الاتجاهين.

unmarked

PromptConvert an RDF/XML snippet into a labelled directed graph. Then convert a graph into RDF/XML. Get the blank-node rules right.

حوّل قطعة RDF/XML إلى رسم بياني موجّه معنون، ثم العكس. وانتبه لقواعد العقد العمياء (blank nodes).

— do part A on paper before peeking.

The three rules you must know cold

  • An RDF statement is a triple: <subject> <predicate> <object>. Subject must be a URI or blank node; predicate must be a URI; object can be URI, blank node, or literal.
  • In RDF/XML: rdf:about="…" sets the subject of a <rdf:Description>; child elements are predicates; their text content is a literal object, while rdf:resource="…" makes the object a URI.
  • Blank nodes are resources with no global URI. They’re indicated by rdf:nodeID="…" (named) or by nesting <rdf:Description> without an about (anonymous). The system assigns an internal id (_:A234) that is not visible outside the file.

Recipe — RDF/XML ↔ Graph in 4 mechanical steps وصفة آلية للتحويل بين RDF/XML والرسم — أربع خطوات لا تحتاج تفكيراً

Don’t try to “understand” it — just follow the recipe. Every RDF/XML document is built from the same 4 patterns. Recognize them and the graph draws itself.

لا تحاول «فهمها» من المبدأ — تتبع الوصفة فقط. كل وثيقة RDF/XML مبنية من ٤ أنماط فقط. تعرّف عليها وسيرسم نفسه.

Direction 1 → from XML to graph

Find each <rdf:Description> block — each one is a NODE. كل بلوك <rdf:Description> يصبح عقدة.

  • rdf:about="X"  ⇒  solid oval, write X inside.
  • rdf:nodeID="X"  ⇒  dashed oval (blank node), write _:X inside.
  • Shorthand: <foaf:Person rdf:about="X"> = same as <rdf:Description rdf:about="X"> + an extra rdf:type foaf:Person triple.

Every child element inside that block = one outgoing ARROW. كل عنصر ابن داخل البلوك يصبح سهماً خارجاً منه.

  • The arrow LABEL is the element name (e.g. f:titre, a:publisher, foaf:name).
  • Where the arrow ENDS depends on what's inside:
    • Plain text >some text<  ⇒  rectangle (literal). Put the text in quotes.
    • rdf:resource="Y"  ⇒  arrow points to oval Y (another URI).
    • rdf:nodeID="Y"  ⇒  arrow points to the dashed oval _:Y (the same blank node you’ll draw from its own block).
    • Nested <rdf:Description> with no about  ⇒  arrow points to an anonymous dashed oval; the children of that inner block become arrows out of it.

Tags & types — small notation on literals. وسوم اللغة وأنواع البيانات تُكتب بجانب القيمة.

  • xml:lang="fr"  ⇒  write @fr next to the literal.
  • rdf:datatype="…integer"  ⇒  write ^^xsd:integer next to the literal.

Connect the pieces. Done. اربط القطع. خلاص.

Same rdf:about="X" appearing in two places ⇒ same oval drawn once. Same rdf:nodeID="X" in two places ⇒ same blank node drawn once. The graph is the union.

Direction 2 → from graph to XML (just reverse)

  1. For each oval on the graph  ⇒  open one <rdf:Description> block.
    • URI oval  ⇒  rdf:about="…"
    • Blank-node oval  ⇒  rdf:nodeID="…" (or no about at all if anonymous)
  2. For each arrow going OUT of that oval  ⇒  add one child element. The arrow label becomes the element name.
    • If arrow → rectangle (literal): put the text inside the element.
    • If arrow → another oval: use rdf:resource="…" (for URI) or rdf:nodeID="…" (for blank node) as an attribute on a self-closing element.
  3. Close the block. Repeat for every oval.

Worked mini-example

Given this 1-block XML:

<rdf:Description rdf:about="http://ex/book1">
  <a:title>Hello</a:title>
  <a:author rdf:nodeID="X1"/>
</rdf:Description>
<rdf:Description rdf:nodeID="X1">
  <a:name>Ali</a:name>
</rdf:Description>

Mechanical translation:

  1. First block → solid oval labelled http://ex/book1.
    • Child a:title with text "Hello" → arrow a:title to a rectangle "Hello".
    • Child a:author rdf:nodeID="X1" → arrow a:author to a dashed oval _:X1.
  2. Second block → the same dashed oval _:X1 (don't draw a new one — same nodeID).
    • Child a:name with text "Ali" → arrow a:name from _:X1 to a rectangle "Ali".

Final graph:

الخلاصة بالعربية:
  1. كل <rdf:Description> = عقدة. لها URI ⟵ بيضوي صلب. لها nodeID ⟵ بيضوي متقطّع (عقدة عمياء).
  2. كل عنصر ابن = سهم خارج من العقدة. اسم العنصر = اسم السهم.
  3. نهاية السهم:
    • نص داخل العنصر ⟵ مستطيل (حرفي).
    • rdf:resource="…" ⟵ بيضوي آخر (URI).
    • rdf:nodeID="…" ⟵ نفس البيضوي المتقطّع المعرّف في بلوك آخر.
  4. اربط القطع. تكرار about أو nodeID في بلوكين ⟵ نفس العقدة مرسومة مرة واحدة.
للاتجاه المعاكس (من رسم إلى XML): العكس بالضبط — كل بيضوي يصير بلوك <rdf:Description>، كل سهم خارج منه يصير عنصراً ابناً.
القواعد الثلاث الأساسية:
  • كل عبارة RDF هي ثلاثية: (subject, predicate, object). الفاعل URI أو عقدة عمياء، المسند URI دائماً، والمفعول URI أو عقدة عمياء أو حرفية (literal).
  • في RDF/XML: rdf:about تحدّد الفاعل، والعناصر الفرعية مسندات، ونصها حرفي، وإن استخدمنا rdf:resource فالمفعول URI.
  • العقدة العمياء (blank node) مورد بلا URI عالمي، تكتب بـ rdf:nodeID أو بتركيب <rdf:Description> بدون about — النظام يولّد لها اسماً داخلياً.

part BGraph → RDF/XML (the “Bassel” example)

You’re given this small graph about a person named Bassel:

subject                       predicate     object
<http://example.org/Bassel>   rdf:type      foaf:Person
<http://example.org/Bassel>   foaf:name     "Bassel ALKHATIB"
<http://example.org/Bassel>   foaf:mbox     <mailto:bassel@svuonline.org>
<http://example.org/Bassel>   ex:age        "40"^^xsd:integer

Step 1 — pick the namespaces

You see three different prefixes: rdf:, foaf:, ex: (and we need xsd: for the typed integer). Declare them on the root <rdf:RDF>.

Step 2 — wrap the subject

You have a choice: use <rdf:Description rdf:about="…"> + an explicit rdf:type, or use the shortcut <foaf:Person rdf:about="…"> (the typed element is the rdf:type).

Step 3 — add the three other triples as children

foaf:name
literal — put text content inside the element.
foaf:mbox
URI — use rdf:resource="…" (empty element).
ex:age
typed literal — element text + rdf:datatype="…#integer" attribute.

Result

<rdf:RDF
   xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#"
   xmlns:foaf="http://xmlns.com/foaf/0.1/"
   xmlns:ex="http://example.org/"
   xmlns:xsd="http://www.w3.org/2001/XMLSchema#">

  <foaf:Person rdf:about="http://example.org/Bassel">
     <foaf:name>Bassel ALKHATIB</foaf:name>
     <foaf:mbox rdf:resource="mailto:bassel@svuonline.org"/>
     <ex:age rdf:datatype="http://www.w3.org/2001/XMLSchema#integer">40</ex:age>
  </foaf:Person>

</rdf:RDF>
كيف تستنتج كود RDF/XML من الرسم؟
  1. أعلِن المسافات (namespaces) في عنصر <rdf:RDF> العام.
  2. افتح <rdf:Description rdf:about="..."> بحيث الـ URI = الفاعل (subject).
  3. اختصار: لو كان لديك rdf:type foaf:Person يمكنك كتابة <foaf:Person rdf:about="..."> مباشرة.
  4. لكل علاقة:
    • المفعول URI ⇒ استخدم rdf:resource="..." (عنصر فارغ).
    • المفعول حرفي ⇒ ضع النص داخل العنصر.
    • حرفي مع نوع بيانات ⇒ أضف rdf:datatype="...".
    • حرفي مع لغة ⇒ أضف xml:lang="fr".

part CThree ways to write a blank node — know all three

Take the “publisher with a name and a city” example. The publisher is a thing with sub-properties — but no global URI. RDF gives you three ways to write it:

Option 1 — invent an external URI (urn:uuid)

<rdf:Description rdf:about="http://…/isbn/000651409X">
   <a:publisher rdf:resource="urn:uuid:f60ffb40-307d-…"/>
</rdf:Description>
<rdf:Description rdf:about="urn:uuid:f60ffb40-307d-…">
   <a:p_name>Harper Collins</a:p_name>
   <a:city>London</a:city>
</rdf:Description>

Option 2 — named blank node (rdf:nodeID)

<rdf:Description rdf:about="http://…/isbn/000651409X">
   <a:publisher rdf:nodeID="A234"/>
</rdf:Description>
<rdf:Description rdf:nodeID="A234">
   <a:p_name>Harper Collins</a:p_name>
   <a:city>London</a:city>
</rdf:Description>

Option 3 — anonymous blank node (nested Description)

<rdf:Description rdf:about="http://…/isbn/000651409X">
   <a:publisher>
      <rdf:Description>
         <a:p_name>Harper Collins</a:p_name>
         <a:city>London</a:city>
      </rdf:Description>
   </a:publisher>
</rdf:Description>
متى تستخدم العقدة العمياء؟ عندما لا يكون للمورد URI عالمي ولا أهمية له خارج هذا الملف (مثلاً «الناشر» عبارة عن كائن داخلي بإسم ومدينة، لا تريد أن يصبح مرجعاً عالمياً).
  • الخيار الأول ⟵ إن أردته مرئياً على الويب، أعطه URI من نوع urn:uuid:.
  • الخيار الثاني ⟵ إن أردت الإشارة إليه في الملف نفسه عدة مرات، أعطه rdf:nodeID.
  • الخيار الثالث ⟵ إن استُخدم مرة واحدة، عشّش rdf:Description فارغ داخل العلاقة.
تحذير عند الدمج: العقد العمياء بنفس الـ nodeID في رسمين مختلفين هي عقد مختلفة — لا تندمج تلقائياً.
Q4 · RDFs

RDF schemas — classes, properties, domain/range, and inference. RDFs deck · slides 1–44 · the “extra knowledge” layer السؤال الرابع: مخططات RDF (RDFs) — الأصناف والخصائص والـ domain/range والاستنتاج التلقائي.

PromptGiven the Bassel example, add an RDFs schema (Person, age, subClassOf Animal) and state what gets inferred. Also explain the MotorVehicle class hierarchy.

انطلاقاً من مثال Bassel، أضف مخطط RDFs (الصنف Person، الخاصية age، علاقة subClassOf مع Animal) واشرح ما الذي يتم استنتاجه تلقائياً.

— inference is the whole point here.

What is RDFs and why do we need it?

RDF gives you triples; RDFs (officially: RDF Vocabulary Description Language) lets you describe what those triples mean: which resources are classes, which are properties, which classes are subclasses of others, what kinds of subjects/objects a property accepts. From this an RDFs reasoner can derive new triples that were never stated.

الفرق بين RDF و RDFs:
  • RDF يعطيك ثلاثيات (subject-predicate-object) — هذه «البيانات».
  • RDFs يضيف طبقة «المعرفة الإضافية» التي تصف معنى البيانات: أصناف (Class)، خصائص (Property)، علاقات تراتبية (subClassOf, subPropertyOf)، ومجال ومدى (domain, range).
  • من هذه الطبقة يستنتج المحرّك (Reasoner) ثلاثيات جديدة لم تكن مذكورة صراحة.

The vocabulary you must know

TermWhat it doesالشرح
rdfs:ClassThe class of all classes.صنف كل الأصناف.
rdfs:ResourceThe class of all resources (everything in RDF).صنف كل الموارد (كل شيء في RDF).
rdfs:LiteralThe class of all literal strings.صنف كل القيم الحرفية.
rdf:PropertyThe class of all properties.صنف كل الخصائص.
rdf:typeSays a resource is an instance of a class.يربط مورداً بصنفه.

part AAdd the schema to the Bassel data

Schema additions: Person is a subclass of Animal · age has domain Person, range Integer.

<rdfs:Class rdf:about="http://xmlns.com/foaf/0.1/Person">
   <rdfs:subClassOf rdf:resource="http://example.org/Animal"/>
</rdfs:Class>

<rdf:Property rdf:about="http://example.org/age">
   <rdfs:domain rdf:resource="http://xmlns.com/foaf/0.1/Person"/>
   <rdfs:range  rdf:resource="http://www.w3.org/2001/XMLSchema#integer"/>
</rdf:Property>

The graph view

What gets inferred

  • Bassel rdf:type Person  +  Person rdfs:subClassOf Animal  ⇒  Bassel rdf:type Animal.
  • Bassel ex:age "40"  +  ex:age rdfs:domain Person  ⇒  Bassel rdf:type Person (redundant — already known).
  • If anyone had written x ex:age "30" without typing x, the rdfs:domain alone would type x as a Person. This is how domain/range inference works.
قاعدتا الاستنتاج اللتان لا تنسى:
  1. التراتب (subClassOf): إن كان A rdf:type B و B rdfs:subClassOf C ⇒ يستنتج النظام A rdf:type C.
  2. المجال والمدى (domain/range): إذا قلنا P rdfs:domain D، فأي مورد يظهر فاعلاً في x P y يصبح تلقائياً من النوع D. وبالمثل للمدى مع المفعول.
تنبيه: domain/range في RDFs لا تمنع القيم المخالفة، بل تستنتج أن الفاعل/المفعول من ذلك النوع. هذا فرق جوهري عن SQL.

5-aMissing separators

PREFIX foaf: <http://xmlns.com/foaf/0.1/>
SELECT ?name ?email
WHERE {
   ?p  foaf:name   ?name
   ?p  foaf:mbox   ?email
}
Incorrect. Triple patterns inside WHERE must be separated by . (period) or ; (semicolon, when the subject is shared). Without them the parser fails.

Fix 1 — one triple per line

SELECT ?name ?email
WHERE {
   ?p foaf:name ?name .
   ?p foaf:mbox ?email .
}

Fix 2 — shared-subject shortcut

SELECT ?name ?email
WHERE {
   ?p foaf:name ?name ;
      foaf:mbox ?email .
}
المثلثات داخل WHERE يجب الفصل بينها بـ . أو بـ ; عند تشارك الفاعل. الكود الأصلي يعطي خطأ نحوي.

5-bVariable missing its ?

PREFIX foaf: <http://xmlns.com/foaf/0.1/>
SELECT person ?name
WHERE {  ?person foaf:name ?name . }
Incorrect. Variables in SPARQL always start with ? (or $). person in the SELECT clause is missing it.

Fix

SELECT ?person ?name
WHERE { ?person foaf:name ?name . }
المتغير في SPARQL يبدأ بـ ? دائماً. person بدون ? ليس متغيراً.

5-cLandlocked countries above 15 M

PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
PREFIX type: <http://dbpedia.org/class/yago/>
PREFIX prop: <http://dbpedia.org/property/>
SELECT ?country_name ?population
WHERE {
   ?country a type:LandlockedCountries ;
            rdfs:label ?country_name ;
            prop:populationEstimate ?population .
   FILTER (?population > 15000000 &&
           langMatches(lang(?country_name), "EN")) .
}
ORDER BY DESC(?population)

Line by line

  • a is shorthand for rdf:type.
  • ; reuses the subject ?country for label and population.
  • FILTER applies the population threshold and picks only the English label (via langMatches(lang(...), "EN")) — otherwise translated duplicates would inflate the result.
  • ORDER BY DESC sorts high→low.
In one sentence: return every landlocked country whose population exceeds 15 million, with its English name and population, sorted from most populous to least.
يعيد كل دولة مغلقة جغرافياً بسكان يتجاوز ١٥ مليون، باسمها الإنكليزي وعدد سكانها، مرتبة تنازلياً بحسب السكان.

5-dAmazon vs Nile — an ASK query

PREFIX prop: <http://dbpedia.org/property/>
ASK {
   <http://dbpedia.org/resource/Amazon_River> prop:length ?amazon .
   <http://dbpedia.org/resource/Nile>         prop:length ?nile   .
   FILTER(?amazon > ?nile) .
}
  • ASK returns true/false, not a table.
  • Binds both lengths and filters “Amazon > Nile”.
  • Returns true iff some binding satisfies the filter.
In words: “Is the Amazon longer than the Nile?” Per DBpedia in the slide → false.
يجيب بـ true أو false. السؤال: هل نهر الأمازون أطول من النيل؟ الجواب وفق DBpedia في الشريحة: false.

5-eMusic artists with OPTIONAL extras

PREFIX foaf: <http://xmlns.com/foaf/0.1/>
PREFIX mo:   <http://purl.org/ontology/mo/>
SELECT ?name ?img ?hp ?loc
WHERE {
   ?a a mo:MusicArtist ;
      foaf:name ?name .
   OPTIONAL { ?a foaf:img       ?img }
   OPTIONAL { ?a foaf:homepage  ?hp  }
   OPTIONAL { ?a foaf:based_near ?loc }
}
In words: list every mo:MusicArtist with their name, plus image / homepage / location if available. Without OPTIONAL, an artist missing any of those would be dropped entirely.
يعيد كل فنان موسيقي مع اسمه، وصورته وصفحته وموقعه الجغرافي إن وُجدت. OPTIONAL تترك العمود فارغاً عند غياب القيمة بدل إسقاط الصف كله.

5-fSampling classes with DISTINCT + LIMIT

SELECT DISTINCT ?concept
WHERE { ?s a ?concept . }
LIMIT 50
In words: iterate every subject, take its class (a = rdf:type), de-duplicate, return 50 — i.e. show 50 example classes that appear in the dataset.
يجلب ٥٠ صنفاً موجوداً في القاعدة كعينة استكشاف. DISTINCT يحذف التكرارات.

Aggregates with GROUP BY (slide 35)

CONSTRUCT — return RDF, not a table (slide 40)

PREFIX vCard: <http://www.w3.org/2001/vcard-rdf/3.0#>
PREFIX foaf:  <http://xmlns.com/foaf/0.1/>
CONSTRUCT {
   ?X vCard:FN    ?name  .
   ?X vCard:URL   ?url   .
   ?X vCard:TITLE ?title .
}
FROM <http://www.w3.org/People/Berners-Lee/card>
WHERE {
   OPTIONAL { ?X foaf:name     ?name  . FILTER isLiteral(?name) }
   OPTIONAL { ?X foaf:homepage ?url   . FILTER isURI(?url) }
   OPTIONAL { ?X foaf:title    ?title . FILTER isLiteral(?title) }
}

Translates a FOAF graph into a vCard graph — vocabulary-to-vocabulary conversion.

CONSTRUCT يستخدم بدلاً من SELECT حين تريد إرجاع رسم RDF جديد بدل جدول. مفيد لتحويل البيانات من ontology إلى أخرى.

  • Tests: isURI, isBlank, isLiteral, bound

  • Other: sameTerm, langMatches, regex

A1RDF Containers — Bag, Seq, Alt

RDF defines three container types when a property has multiple values:

  • rdf:Bag — an unordered list (e.g. authors of a book — order doesn’t matter).
  • rdf:Seq — an ordered list (e.g. students alphabetically).
  • rdf:Alt — a list of alternatives (only one is meant — e.g. file formats: html, plain, postscript).

Items inside are listed as <rdf:li>...</rdf:li>.

<!-- Bag: the authors of the book 0201000237 are Alfred, John and Jeffrey -->
<rdf:Description rdf:about="urn:ISBN:0-201-00023-7">
  <dc:creator>
    <rdf:Bag>
      <rdf:li>Alfred</rdf:li>
      <rdf:li>John</rdf:li>
      <rdf:li>Jeffrey</rdf:li>
    </rdf:Bag>
  </dc:creator>
</rdf:Description>
الحاويات الثلاث:
  • Bag: قائمة غير مرتبة — مثل مؤلفي كتاب.
  • Seq: قائمة مرتبة — مثل ترتيب الطلاب أبجدياً.
  • Alt: قائمة بدائل (واحد منها فقط مقصود) — مثل صيغ ملف PDF/HTML/TXT.
عناصرها تكتب بـ <rdf:li>. تذكّر: حاوية واحدة فقط من النوع الصحيح بحسب دلالتك.

Container vs repeated property

Two different meanings — important difference:

  • Repeated property: “Sue has written A, B, and C” — three separate triples, no claim that they relate to each other.
  • Container: “the committee of Fred, Wilma, Dino approved the resolution” — the group as a whole did it, not each member individually.

A2parseType — when to use "Resource" vs "Literal"

rdf:parseType="Resource" — lets you describe a blank node inline by listing its sub-properties as children, without wrapping in an extra <rdf:Description>. Use it when you need a small structured value (price + currency, address, etc.).

<!-- The cost of the book 0201000237 is $45.00 -->
<rdf:Description rdf:about="urn:ISBN:0-201-00023-7">
  <s:price rdf:parseType="Resource">
     <rdf:value>45.00</rdf:value>
     <s:units rdf:resource="http://…/USdollar"/>
  </s:price>
</rdf:Description>

rdf:parseType="Literal" — tells the RDF parser not to interpret the content as RDF. Use it when the object is embedded XML/MathML that should be kept verbatim.

<dc:title rdf:parseType="Literal">
  Solving <mrow><msup><mi>x</mi><mn>2</mn></msup><mo>=</mo><mn>0</mn></mrow>
</dc:title>
parseType:
  • "Resource" ⟵ يخبر المحلّل أن المحتوى الداخلي خصائص مورد (blank node) — مفيد لاختصار وصف قيمة منظّمة كالسعر + العملة.
  • "Literal" ⟵ يخبره أن لا يحلّل المحتوى الداخلي كـ RDF — مفيد لتضمين XML خام أو MathML.

A3Reification — statements about statements

Sometimes you want to say something about a statement, not about the resource. Example:

  • “Ora Lassila is the creator of …/Home/Lassila” — a plain statement.
  • “Ralph Swick says that Ora Lassila is the creator of …/Home/Lassila” — a statement about the first statement.

RDF gives four reification properties to model the inner statement as a resource of its own (an instance of rdf:Statement):

  • rdf:subject — the original subject
  • rdf:predicate — the original predicate
  • rdf:object — the original object
  • rdf:type = rdf:Statement

Then you can hang attributes on it (like a:attributedTo "Ralph Swick").

<rdf:Description>
  <rdf:type rdf:resource="…#Statement"/>
  <rdf:subject   rdf:resource="http://www.w3.org/Home/Lassila"/>
  <rdf:predicate rdf:resource="…#Creator"/>
  <rdf:object>Ora Lassila</rdf:object>
  <a:attributedTo>Ralph Swick</a:attributedTo>
</rdf:Description>
Reification (التصنيم): الطريقة في RDF لقول شيء عن عبارة، لا عن مورد. ننشئ مورداً جديداً من النوع rdf:Statement ونعطيه أربع خصائص: rdf:subject, rdf:predicate, rdf:object, و rdf:type. ثم نلصق به أي معلومات إضافية (مثل من قاله ومتى).

A4RDFs — using rdfs:range to constrain to a small set

A common RDFs trick: declare a class MaritalStatus and four named instances (Single, Married, Divorced, Widowed). Then declare a property maritalStatus with rdfs:range MaritalStatus. Now the value can only sensibly be one of the four.

<rdfs:Class rdf:ID="MaritalStatus"/>

<rdf:Property rdf:ID="maritalStatus">
  <rdfs:domain rdf:resource="#Person"/>
  <rdfs:range  rdf:resource="#MaritalStatus"/>
</rdf:Property>

<rdf:Description rdf:ID="Married">
  <rdf:type rdf:resource="#MaritalStatus"/>
</rdf:Description>
<!-- ...same for Single, Divorced, Widowed -->
الفكرة: أنشئ صنفاً MaritalStatus، ثم أربعة موارد من هذا الصنف (متزوج، أعزب، مطلّق، أرمل). ثم اضبط rdfs:range للخاصية maritalStatus ليكون هذا الصنف. الآن أي شخص لا يمكن أن تكون حالته الاجتماعية إلا واحدة من الأربعة المُعرَّفة.

A5Non-binary relations — use an intermediate blank node

RDF only has binary relations (between two things). To express ternary or higher (e.g. “price = 45 in USD”), introduce an intermediate resource:

<s:price rdf:parseType="Resource">
  <rdf:value>45.00</rdf:value>
  <s:units rdf:resource="…/USdollar"/>
</s:price>

The blank node is the price, and it has both a numeric value and a currency.

A6Dublin Core — 15 standard properties for any web resource

An ISO standard (15836-2003) vocabulary, namespace http://purl.org/dc/elements/1.1/. Used to describe published content (documents, datasets).

The 15:

  1. Title
  2. Creator
  3. Subject
  4. Description
  5. Publisher
  6. Contributor
  7. Date
  8. Type
  9. Format
  10. Identifier
  11. Source
  12. Language
  13. Relation
  14. Coverage
  15. Rights
Dublin Core: معجم معياري (ISO 15836) بـ ١٥ خاصية لوصف الموارد المنشورة. مفيد لميتاداتا الوثائق والكتب. النطاق هو http://purl.org/dc/elements/1.1/.

vCard — for people

What Dublin Core is for documents, vCard is for people (IETF standard). Properties: FN (formatted name), N (structured name with Given/Family), EMAIL, URL, ORG, TITLE, etc. Namespace http://www.w3.org/2001/vcard-rdf/3.0#.

A7The motivating task — dog | frisbee | catch | throw

Common Sense deck opens with this: given four concepts, an AI must produce a meaningful sentence. To do that it needs all the implicit relations a human knows:

  • dog — capableOf — play game
  • frisbee — typeOf — flying disk
  • catch — antonymOf — throw
  • PersonX throws a frisbee — asA result — others catch frisbee (ATOMIC-style inference)
  • fun for dog — has property — play frisbee (ConceptNet-style triple)

This is the canonical example used to motivate why CSKGs exist — language models alone can’t reliably make these leaps.
